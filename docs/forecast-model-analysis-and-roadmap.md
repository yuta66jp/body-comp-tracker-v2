| NeuralProphet | NP | 学習ベースの時系列モデル。GitHub Actions 日次バッチで実行 |

## 3. モデル別評価

### 3-1. Linear Trend 30d — 短中期の王者、長期では崩壊

**強み:**
- D+7 / D+14 で単日評価の MAE 最良
- 通常日（イベント除外）では全 horizon で最良
- MAPE が全モデル中最小（D+7: 0.94%, D+14: 1.42%）

**弱み — Bias の加速的悪化:**

| Horizon | Bias | 解釈 |
|---|---|---|
| D+7 | -0.147 | 軽微な下振れ（許容範囲） |
| D+14 | -0.635 | 実測より 0.6 kg 低く予測する傾向 |
| D+30 | **-1.292** | 実測より 1.3 kg 低く予測 — 外挿の暴走 |

**原因**: 30日間の線形トレンドをそのまま外挿するため、実際の体重変動（減速・水分回復・リバウンド）を拾えず、直線的に下がり続ける予測を出す。horizon が長いほどこの乖離が拡大する。

**D+30 での急落の追加要因**: イベント除外評価で D+30 MAE が 1.613 → 0.697 に劇的改善することから、D+30 の悪化の大半はチートデイ期間（1/28〜3/9）の影響。クリーンデータが蓄積されれば D+30 の実力値は大幅に改善する可能性がある。

**RMSE/MAE 比:**

| Horizon | MAE | RMSE | RMSE/MAE |
|---|---|---|---|
| D+7 | 0.667 | 0.852 | 1.28 |
| D+14 | 1.016 | 1.323 | 1.30 |
| D+30 | 1.613 | 2.080 | 1.29 |

比率が約1.3で一定 = 外れ値的な巨大誤差は少なく、誤差分布は比較的均一。ただし D+30 の RMSE 2.08 kg は個別予測で 2 kg 超の誤差が常態化していることを意味する。

### 3-2. Naive — 長期の安定王

**強み:**
- D+30 の単日評価・7日平均評価の両方で MAE 最良
- Bias が全 horizon で +0.1〜+0.3 kg と安定（系統的な大外れがない）
- 最もシンプルで実装コスト・計算コストがゼロ

**弱み:**
- D+7 / D+14 では下位（直近のトレンドを反映できない）
- 7日平均評価の D+7 で 1位になるが、これは SMA7 自体が平滑化済みのため Naive との相性が良い構造的バイアス

**Naive が D+30 で勝つ理由**: 他モデルが「トレンドを外挿しすぎる」のに対し、Naive は「現在地をそのまま返す」だけなので、外挿暴走がない。D+30 のような長期では「何もしない」方が「間違った方向に外挿する」より良い結果になる。

### 3-3. EW Linear Trend — バランス型

**強み:**
- 全 horizon で 2〜3 位に安定
- 指数加重により直近の変化に対する感度が Linear 30d より高い
- Bias の暴走も Linear 30d ほどではない（D+30: -0.904 vs -1.292）

**弱み:**
- どの horizon でも1位を取れない
- Bias は Linear 30d と同方向（負 = 下振れ傾向）で、程度が軽いだけ

**位置づけ**: Linear 30d と Naive の中間的な特性。アンサンブルの構成要素として有望。

### 3-4. NeuralProphet — 期待外れ

**強み:**
- D+7 で 2位（Linear 30d に僅差）
- 理論上はイベント・季節性・外部変数を組み込める拡張性

**弱み:**
- D+14: 4位、D+30: 最下位
- n=14 / 168日分のデータでは学習ベースモデルの真価を発揮できない
- epochs=100 でオーバーフィットの懸念

**現時点の評価**: data-hungry なモデルに投資するより、Linear 系の補正に注力すべき段階。半年〜1年のデータ蓄積後に再評価。

### 3-5. MA 7d — 中位の安定選手

- 全 horizon で中位（3〜5位）
- Bias が一貫して正（+0.3〜+0.5）= 常にやや高めに予測
- 特別な強みはないが、アンサンブルの構成要素としては使える

### 5-3. Phase 2 でのバックテスト再実行の具体手順

```bash
# 1. D+60/D+90 を追加した horizons で実行
python ml-pipeline/backtest.py \
  --series-type daily \
  --horizons 7 14 30 60 90 \
  --max-origins 20 \
  --origin-step-days 5

# 2. SMA7 系列でも同様に実行
python ml-pipeline/backtest.py \
  --series-type sma7 \
  --horizons 7 14 30 60 90 \

  --origin-step-days 5
```

**判断基準:**
- D+60 の Linear 30d Bias が -2.0 を超える → 長期では Naive or EWLinear を採用
- D+60 の Linear 30d Bias が -1.0 以内 → クリーンデータではトレンド外挿が有効と判断
- 全 horizon で特定モデルが安定的に最良 → 動的選択不要、単一モデルで十分

### 5-4. 動的 Horizon 別モデル選択の設計方針（Phase 3）

大会までの残り日数に応じて、参照する horizon 帯を自動切替する。

```
残り日数 → 参照 horizon → 採用モデル（Phase 2 の結果で確定）

残り 90日以上: D+60/D+90 のバックテスト結果で選択
残り 60〜90日: D+30 の最良モデル
残り 30〜60日: D+14 と D+30 の加重
残り 30日未満: D+7 / D+14 の最良モデル
```

**実装場所**: ダッシュボードの「目標到達予定日」計算ロジック。
現在の「7日平均 + 30日トレンド」を、バックテスト結果駆動に切り替える。

## 6. ダッシュボード・各画面の改善提案

### 6-1. Phase 1 の実施内容（2026-03-29 完了）

Phase 1 の施策①〜④はすべて実装・マージ済み。以下は各施策の実施内容と現在の状態。

#### ① 到達予測の計算透明化 ✅（#397）

**実装前の状態:**
- KPI カード: 「目標到達予定 2026-08-30 / 58.0 kg 到達の推定日（7日平均 + 30日トレンド）」
- GoalNavigator でバッファ日数を表示しようとして、到達予測系の計算が GoalNavigator と page.tsx に分散していた

**実装内容:**
- `page.tsx` を `goalReachResult` / `bufferDays` の単一算出点に統一
- バッファ日数（「+N 日の余裕」/「▲N 日不足」）を KPI カードの「目標到達予定」セルに表示
- GoalNavigator は従来の役割（現在体重/目標/残り/ペース/調整）に戻し、到達予測系の補足表示を持たせない設計に整理
- 計算の重複排除により GoalNavigator 内の `toJstDateStr()` 再呼び出しも解消

**現在の状態:**
- KPI カード「目標到達予定」セルにバッファ日数が色分き表示（緑: +14日以上 / 黄: 0〜13日 / 赤: 不足）
- GoalNavigator のペース分析（kg/2週が primary 単位）と重複なし

#### ② %BW/週の推移トラッキング ✅（#402 / #403）

**実装前の状態:**
- WeeklyReview に「週次体重変化」の kg 値は表示されていたが Helms 基準との照合がなかった

**実装内容（#402）:**
- `calcWeeklyReview.ts` の `WeeklyWeight` に `bwRatePctPerWeek: number | null` フィールドを追加
  - 計算式: `-(trendKgPerWeek / avgWeight) × 100`（14日線形回帰 ÷ 7日平均体重）
- `WeeklyReviewCard.tsx` に BWRateStatus ロジック（gaining / very_slow / slow / recommended / slightly_fast / too_fast）と表示を追加

**実装内容（#403）:**

## 8. 実装時の注意事項

### 8-1. CLAUDE.md の遵守事項（本ロードマップに関連するもの）

| 原則 | Phase 1 での具体的な注意 |
|---|---|

| 同じ意味の値を別ロジックで増やさない | ② %BW/週は GoalNavigator のペース分析（kg/2週）と意味が異なることを確認済み |

### 8-2. データ品質に関する前提

- 「クリーンデータ」= チートデイ・旅行・手動イベントフラグが付いていない期間（2026-03-10 以降）
- Phase 2 のバックテスト再実行では `data_start_date` パラメータで 3/10 以降に限定可能にしておく
- 現在の除外対象（44日 / 168日 = 26%）は多く、通常日評価の信頼性に影響している
- フラグの粒度見直し（回復期間の長さ = 現状3日）は運用期間中の観察で判断

### 8-3. バックテストサンプル数の見積もり

```
Phase 2 時点（5月初旬）:
  クリーンデータ: 約55日（3/10〜5/5）
  step=5d, max-origins=10 とすると:
    D+7:  10 サンプル（55 - 7 = 48日 / 5 ≒ 9.6）
    D+14: 8 サンプル
    D+30: 5 サンプル
    D+60: 0〜1 サンプル（ギリギリ）

Phase 3 時点（6月）:
  クリーンデータ: 約85日（3/10〜6/5）
  step=5d, max-origins=15 とすると:
    D+7:  15 サンプル
    D+14: 14 サンプル
    D+30: 11 サンプル
    D+60: 5 サンプル
```

D+60 のサンプル数が実用的になるのは Phase 3（6月）以降。Phase 2 では D+60 は参考程度の扱い。

## 付録

### A. 用語集

| 用語 | 定義 |
|---|---|
| MAE | Mean Absolute Error。予測と実測の平均絶対誤差（kg） |
| RMSE | Root Mean Squared Error。大きな誤差にペナルティがかかる指標 |
| MAPE | Mean Absolute Percentage Error。誤差の体重比（%） |
| Bias | 平均誤差（予測 − 実測）。正=高めに予測、負=低めに予測する傾向 |
| horizon | 予測の先読み日数（D+7 = 7日先予測） |
| walk-forward | 時系列データで過去のみを使って学習し未来を予測するバックテスト手法 |
| origin | バックテストの予測起点。step 間隔で複数設定される |
| クリーンデータ | チートデイ・旅行・手動イベント期間を除いた通常記録日のデータ |
| batch canonical | `enrich.py` / `analyze.py` 等のバッチ処理が算出した値を唯一の正とする設計方針 |
| %BW/週 | 週あたりの体重変化を体重比で表した値（例: 70kg → 69.2kg = -1.14%） |

### B. 参考文献

| 文献 | 要旨（本プロジェクトとの関連） |
|---|---|
| Helms et al. (2014) | ナチュラルボディビルダーの減量ガイドライン。推奨減量ペース 0.5〜1.0% BW/週 |

| Rossow et al. (2013) | コンテスト準備期間の代謝・ホルモン変化の追跡研究 |
